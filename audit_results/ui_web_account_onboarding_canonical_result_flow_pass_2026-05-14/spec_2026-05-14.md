# Spec: UI Web Account Onboarding Canonical Result Flow

## Objective

Recompose the web-design account onboarding surface into a safe request modal
and result flow. The UI must show reserve-first success only when the
`onboard_account` command packet proves it, and it must keep no-new-auth,
ambiguous, validation-failed, status-proof-failed, command-error, timeout, and
invalid-JSON outcomes out of the green success path.

## In Scope

- Rework the onboarding request modal into a reserve-first command request.
- Rework the onboarding result flow into a full result surface.
- Gate selected backend display behind proven reserve-first success.
- Add explicit status proof, refresh state, pool state, and non-success copy.
- Preserve the browser boundary: no file input, no path input, no secret input.
- Add/adjust web-design UI tests for the success and non-success paths.
- Capture screenshots and metrics for modal/result states at 1600x1000.

## Out of Scope

- Runtime behavior changes.
- Command adapter changes or allowlist expansion.
- Desktop/native file picker or bridge work.
- Import-existing account flow.
- Promote/demote/retire lifecycle implementation.
- Direct config, auth file, raw log, or state reads.

## Constraints

- Command packet truth remains the only source of onboarding success.
- Browser payload must not contain arbitrary path, auth, secret, or backend id
  values for onboarding.
- Reserve-first success requires proof, not optimistic UI inference.
- Onboarding success never means active routing.
- Visual work must not touch execution-core or desktop files.

## Acceptance Criteria

- [x] Request modal says web does not accept paths, read files, or pass secrets.
- [x] Result flow shows selected backend only for proven reserve-first success.
- [x] Green result requires selected backend, reserve proof, reserve pool, no
  active routing change, validation OK, and status proof OK.
- [x] No-new-auth and ambiguous outcomes are non-green and hide selected backend.
- [x] Validation/status proof/command/invalid-JSON failures are non-green.
- [x] Modal/result fit 1600x1000 without inputs, visible SVG icons, or overflow.
- [x] UI tests cover success, ambiguous, non-success, and contradicted proof.
