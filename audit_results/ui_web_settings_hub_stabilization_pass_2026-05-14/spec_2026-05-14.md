# Spec: UI_WEB_SETTINGS_HUB_STABILIZATION_PASS

## Objective

Convert the web Settings screen from a long settings form into a compact read-only hub with eight canonical sections, clear deferred controls, and no settings mutation surface.

## In Scope

- Settings screen HTML structure.
- Settings-specific CSS hub layout.
- Settings banner state copy.
- UI tests that pin the read-only hub contract.
- Baseline and after screenshots plus 1600x1000 DOM metrics.

## Out of Scope

- Runtime changes.
- Command adapter changes.
- Live server changes.
- Desktop changes.
- Config writes.
- Path pickers or file inputs.
- Secret/token/password entry or value display.
- Recovery or repair execution.

## Constraints

- Settings must not contain `data-ui-action`.
- Settings must not show visible inline SVG icons.
- Settings must not expose config-write, arbitrary path, raw secret, token, password, or direct file-read controls.
- Links from Settings may only navigate to existing screens.
- Save/cancel must remain absent unless a future admitted dirty-save surface exists.

## Assumptions

- Settings can show observed runtime fields that are already populated from bounded packets.
- Existing `settingsLaunchAvailability` remains a read-only availability label.
- API Routes and Diagnostics links are navigational, not command dispatch.

## Acceptance Criteria

- [x] Settings has exactly eight `data-settings-card="true"` cards.
- [x] Settings main content fits inside 1600x1000 without scroll spill.
- [x] `settingsScreen.bottom <= 934`.
- [x] `main.scrollHeight <= main.clientHeight + 1`.
- [x] `visibleSvgIcons === 0` inside Settings.
- [x] `visibleImgIcons > 0` inside Settings.
- [x] `activeSettingsActions === 0`.
- [x] No button text clipping and no fact overflow.
- [x] Live integration failure copy does not imply saved state.
