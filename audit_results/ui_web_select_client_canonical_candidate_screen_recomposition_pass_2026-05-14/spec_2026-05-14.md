# Spec: UI Web Select Client Canonical Candidate Screen Recomposition

## Objective

Recompose the `Select Client` web screen into a safe candidate-selection
surface. The screen must look like a real first-run client selection step while
preserving the web boundary: browser UI may display command-owned candidates,
but must not accept arbitrary local paths, open file pickers, scan the file
system, write configuration, or claim runtime readiness.

## In Scope

- Rework `selectClientScreen` markup into a first-run stepper plus candidate
  list, selection preview, manual-picker deferred block, and bottom action bar.
- Add screen-specific banner copy for fixture, stale fixture, and live failure.
- Keep candidate paths as inert display text with `title` attributes.
- Keep all candidate selection, check, and save controls disabled until a
  future admitted command-owned candidate packet exists.
- Add CSS geometry for stable `1600x1000` fit and Phosphor image icons.
- Extend web-design tests around the select-client boundary.
- Capture baseline and after screenshots/metrics for fixture healthy,
  degraded, down, stale, and live integration failure states.

## Out of Scope

- Runtime, command adapter, allowlist, live server, desktop bridge, native file
  picker, filesystem discovery, installer work, and config writes.
- Real client validation, real save dispatch, client launch, route mutation, and
  import-existing behavior.
- Accounts, Diagnostics, Settings, Setup, and API Connections redesign work.

## Constraints

- `selectClientScreen` must contain no `<input>`, `<select>`, `type="file"`,
  `data-ui-action`, or `live-action`.
- No browser payload keys for `client_path`, `source_dir`, `source-dir`,
  `auth_ref`, secrets, or passwords.
- Manual picker must be represented as desktop/native deferred, not as a browser
  file chooser.
- Search affordance is local-filter copy only, not disk discovery.
- Visible select-client UI icons must be PNG image icons, not inline SVG.
- Live failure must not reuse fixture candidates as discovered local truth.

## Assumptions

- Fixture candidates are acceptable for visual preview when explicitly labelled
  as fixture display.
- The global sidebar preview controls may remain outside the screen body.
- The only active button allowed in this contour is the already existing close
  control.

## Acceptance Criteria

- [x] `selectClientScreen` is tagged as `data-select-client-mode="candidate-preview"`.
- [x] Header and banner copy describe bounded candidates, not filesystem discovery.
- [x] Candidate list has stable rows, status chips, inert paths, and disabled
  candidate buttons.
- [x] Manual picker is desktop/native deferred with browser path payload
  explicitly forbidden.
- [x] Check and save controls are disabled and explain the missing proof/surface.
- [x] Metrics at `1600x1000` show no overflow and `screenBottom <= 934`.
- [x] Metrics report `visibleSvgIcons: 0` for the rendered screen.
- [x] Tests cover the no-input, no-action, no-save, and no-path-payload boundary.
