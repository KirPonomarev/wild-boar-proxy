# Spec: Diagnostics History Bounded Packet Admission

## Objective

Admit a future `Diagnostics Live History` surface only through a bounded,
redacted JSON packet. This contour does not implement the UI, does not add or
admit a command, and does not change runtime, command adapter, allowlist, live
server, or desktop/native surfaces.

The purpose is to prevent a diagnostics chart or records list from becoming a
false runtime-truth surface sourced from raw logs, cached state, diagnostics
bundles, or fixture/demo data.

## In Scope

- Define the minimum bounded diagnostics history packet shape.
- Define freshness, TTL, redaction, allowed fields, and forbidden fields.
- Define the future UI semantics that may be claimed from the packet.
- Define the future visual constraint for the telemetry scale without
  implementing CSS, HTML, or JavaScript.
- Define admission states and color semantics.
- Review current web UI and live-server surfaces for raw log/state/bundle reads.
- Preserve machine-readable admission artifacts under `audit_results/`.

## Out of Scope

- Implementing `Diagnostics Live History` UI.
- Creating or admitting a new command surface.
- Changing `COMMAND_API.md`, runtime, command adapter, allowlist, live server, or
  desktop/native code.
- Reading raw logs, state files, or diagnostics export bundles.
- Parsing support bundles in UI.
- Adding repair, rollout, or healthcheck claims.

## Packet Contract

Candidate future command contract:

```text
diagnostics history --json
implemented_now: false
admitted_now: false
```

The command name above is a future contract placeholder only. It is not admitted
by this contour and must not be wired to UI before a later command-admission
contour.

Minimum packet:

```json
{
  "schema_version": 1,
  "status": "ok",
  "observed_at": "2026-05-15T18:30:00Z",
  "ttl_seconds": 120,
  "window": {
    "label": "last_48_minutes",
    "bucket_seconds": 60,
    "bucket_count": 48
  },
  "summary": {
    "success_count": 42,
    "failure_count": 6,
    "stale_count": 3,
    "unknown_count": 0
  },
  "series": [
    {
      "timestamp": "2026-05-15T18:01:00Z",
      "state": "success",
      "count": 1
    }
  ],
  "records": [
    {
      "timestamp": "2026-05-15T18:28:54Z",
      "object_type": "account",
      "object_ref": "acc-021",
      "severity": "error",
      "machine_code": "AUTH_UNAVAILABLE",
      "message": "auth unavailable"
    }
  ],
  "redaction": {
    "paths_redacted": true,
    "secrets_redacted": true,
    "raw_logs_included": false
  },
  "next_action": "retry"
}
```

Allowed `status` values:

- `ok`
- `empty`
- `stale`
- `unavailable`
- `error`

Allowed `series[].state` values:

- `success`
- `failure`
- `stale`
- `unknown`

Allowed `records[].object_type` values:

- `account`
- `route`
- `sync`
- `auth`
- `upstream`

Allowed `records[].severity` values:

- `ok`
- `warning`
- `error`

Allowed `next_action` values:

- `retry`
- `export_diagnostics`
- `none`

## Freshness Rules

- `observed_at + ttl_seconds` defines packet freshness.
- A packet older than its TTL must render as stale even when `status` is `ok`.
- Missing `observed_at`, missing `ttl_seconds`, invalid timestamp, or invalid TTL
  is a critical missing-fields condition.
- `status=ok` means the diagnostics history packet parsed, passed redaction, and
  is fresh. It does not mean runtime is healthy.
- `status=empty` is a valid bounded packet result and must not be rendered as a
  hard failure.
- `status=unavailable` means no history surface is available; prior fixture or
  healthy data must not be reused as live history.

## Allowed Fields

The future packet may contain:

- bounded timestamps;
- bucketed counts;
- stable machine codes;
- redacted object references;
- short sanitized messages;
- window metadata;
- freshness metadata;
- redaction booleans;
- bounded enum next action.

## Forbidden Fields

The future packet must not contain:

- raw log lines;
- absolute file paths;
- auth tokens, API keys, secret values, or secret material;
- raw command argv;
- stack traces;
- diagnostics bundle contents;
- browser-provided paths;
- raw command packets;
- unredacted personal identifiers unless separately admitted by the relevant
  account/route surface;
- claims such as `runtime healthy`, `account usable`, `route active`,
  `repair needed`, or `rollout proven`.

## UI Semantics

Future UI may show:

- packet status;
- packet freshness;
- bucketed success/failure/stale/unknown counts;
- latest sanitized records;
- bounded history window label;
- redaction status.

Future UI must not claim:

- runtime health;
- account usability;
- route activity;
- repair requirement;
- rollout proof;
- support bundle truth.

The label must make the scope explicit, for example:

```text
Diagnostics history · bounded packet
```

## Future Visual Constraint

The future graph should be a wide operational telemetry scale, not a decorative
sparkline:

- 48 small vertical ticks;
- green for `success`;
- red for `failure`;
- amber for `stale`;
- muted for `unknown` or no data;
- full detail-column width;
- counters and legend below;
- no claim beyond the bounded packet.

No CSS, HTML, JavaScript, screenshot, or design implementation is admitted in
this contour.

## State Matrix Requirements

The admission matrix must define these states:

- `ok`
- `empty`
- `stale`
- `unavailable`
- `invalid_json`
- `missing_fields`
- `timeout`
- `integration_failure`
- `redaction_failed`

Color semantics:

- green only for a fresh `status=ok` packet;
- amber for `empty`, `stale`, `unavailable`, `timeout`, or partial/needs-action
  conditions;
- red for invalid JSON, critical missing fields, integration failure, or
  redaction failure;
- neutral for no selected packet or unknown/no-data placeholders.

## Acceptance Criteria

- [x] Bounded packet boundary is specified.
- [x] Future command remains explicitly not implemented and not admitted.
- [x] Field rules and forbidden fields are explicit.
- [x] UI semantics do not imply runtime truth.
- [x] Future visual constraints are stated without implementation.
- [x] Current web UI/live-server scan is preserved as an artifact.
- [x] Runtime, command adapter, allowlist, live server, and desktop/native files
  are not changed by this contour.

## Verification

- tests: existing web UI/live-server/command-adapter unit tests.
- build: JSON artifacts validate with `python3 -m json.tool`.
- manual: source scan reviewed for raw log/state/bundle reads.
- live evidence: not applicable to this admission contour; no live runtime
  command is admitted or executed.

## Open Questions

- A later command-admission contour must decide whether the future packet is
  provided by `diagnostics history --json` or by another command-owned surface.
- A later implementation contour must decide whether diagnostics history lives
  inside `Diagnostics` or behind a dedicated drawer/panel.
