# Wild Boar Proxy Master Plan

PLAN_VERSION: 2.0
PLAN_DATE: 2026-05-16
PLAN_STATUS: fast product path

## Главная цель

Сделать рабочий управляющий продукт быстро, дёшево и надёжно:

1. Базовый веб уже есть.
2. Дальше подключаем безопасные действия.
3. Проверяем их прямо через веб, как пользователь.
4. Только потом доделываем дизайн и переносим проверенное в настольное
   приложение.

Главный принцип: каждый контур должен давать видимый пользовательский прогресс.
Если контур не приближает кнопку, сценарий, проверку или сборку, он не нужен.

## Канон

Когда документы конфликтуют, порядок такой:

1. `CANON.md`
2. `MASTER_PLAN.md`
3. `RUNTIME_CONTRACT.md`
4. `STATE_SCHEMA.md`
5. `COMMAND_API.md`
6. `DELIVERY_RULES.md`
7. `README.md`

Канон нужен как защита от вранья и поломки рабочего окружения. Он не должен
превращаться в бюрократию.

## Нельзя ломать

- Нельзя трогать текущий рабочий экземпляр Codex, через который идёт работа.
- Нельзя выдавать красное или неизвестное состояние за зелёное.
- Нельзя запускать опасные действия без понятного владельца, проверок и
  результата.
- Нельзя коммитить личные данные, токены, рабочие runtime-файлы, логи и дампы.
- Нельзя делать stage/pilot/scale claims без машинных доказательств.
- Нельзя повторять одну и ту же упавшую проверку по кругу. Два одинаковых
  отказа подряд означают: стоп, локализовать причину, затем идти дальше.

## Что уже считаем сделанным

- Базовый веб-интерфейс создан.
- Безопасный read-only слой в вебе создан:
  - статусы;
  - причины блокировок;
  - журналы команд;
  - сводка текущей правды.
- Живые опасные действия пока не должны быть доступны как обычные кнопки.
- Повторный `LOCK_HELD` локализован до общей процедуры:
  `run_stable_runtime_launcher_attempt`.
- Следующий технический блокер: безопасно развести запуск/проверку runtime и
  selector refresh, чтобы они не дрались за один замок.

## Рабочий режим

Делаем короткими проходами.

Один проход:

1. Одна пользовательская цель.
2. Один маленький кусок кода.
3. Один автотест по изменённому месту.
4. Один браузерный прогон, если это веб-сценарий.
5. Один коммит и push.

По умолчанию:

- без дорогих агентов;
- без больших расследований;
- без лишних артефактов;
- один исполнитель;
- один аудитор только перед закрытием рискованного изменения.

Большое расследование разрешено только если есть риск:

- сломать текущий Codex;
- испортить реальные аккаунты;
- записать runtime-мусор в репозиторий;
- получить ложный зелёный статус;
- смешать разные подсистемы в один контур.

## Проверка через веб

Для веб-функций главный путь проверки такой:

1. Поднять локальный веб.
2. Открыть его браузером.
3. Нажать нужную кнопку.
4. Проверить текст, состояние, результат и отсутствие ложного зелёного.
5. Зафиксировать короткий отчёт.

Если агент может сам нажать кнопку и проверить результат, так и делаем.
Не заменяем пользовательскую проверку длинной теорией.

## Защита текущего Codex

Запуск приложения из веба разрешён только как отдельная копия:

- отдельный профиль;
- отдельная папка данных;
- отдельный порт;
- отдельный процесс;
- текущий рабочий Codex не трогать.

Первый запуск делаем через сухой режим или безопасную заглушку. Реальный запуск
разрешён только после того, как путь, профиль, порт и изоляция явно проверены.

## Контуры

### 1. `RUNTIME_LAUNCHER_OWNER_PROCEDURE_SERIALIZATION_FIX_PASS`

Цель: убрать драку за общий замок между запуском/проверкой runtime и selector
refresh.

Зачем: без этого безопасные действия будут снова упираться в `LOCK_HELD`, и мы
вернёмся к тупому повтору проверок.

Делаем только:

- минимальное изменение процедуры `run_stable_runtime_launcher_attempt`;
- защиту от пересечения с selector refresh;
- тест на отсутствие повторного self-block;
- без UI, дизайна, аккаунтов, stage/pilot.

Результат: можно честно двигаться к безопасным веб-кнопкам.

### 2. `WEB_SAFE_APP_COPY_LAUNCH_PASS`

Цель: в вебе добавить кнопку запуска отдельной копии приложения.

Зачем: это первый реальный пользовательский сценарий управления, при этом он не
должен трогать текущий рабочий Codex.

Делаем:

- кнопку в вебе;
- проверку пути запуска;
- отдельный профиль/папку/порт;
- сухой режим;
- затем реальный запуск, если безопасно;
- браузерный автоклик;
- проверку, что текущая сессия не затронута.

Результат: пользователь может из веба безопасно запустить отдельную копию.

### 3. `WEB_SAFE_ACCOUNT_CONNECT_DRY_RUN_PASS`

Цель: подготовить подключение аккаунтов через веб без реального изменения
аккаунтов.

Зачем: сначала проверяем форму, валидацию, ошибки и результат, не трогая живые
данные.

Делаем:

- форму подключения;
- проверку входных данных;
- сухой результат;
- понятные ошибки;
- браузерный прогон.

Результат: сценарий подключения понятен и безопасен.

### 4. `WEB_SAFE_ACCOUNT_CONNECT_LIVE_PASS`

Цель: включить реальное подключение аккаунта через веб.

Зачем: это ключевая продуктовая функция.

Делаем только после сухого режима:

- явное подтверждение действия;
- declared write surfaces;
- rollback/отказ при ошибке;
- машинный результат;
- браузерная проверка.

Результат: аккаунт можно подключить через веб без ручной возни.

### 5. `WEB_SAFE_COMMANDS_EXPANSION_PASS`

Цель: добавить остальные безопасные веб-действия.

Зачем: собрать рабочую панель управления перед дизайном.

Кандидаты:

- обновить статус;
- посмотреть аккаунты;
- открыть папку профиля;
- экспортировать диагностику без личных данных;
- показать причину блокировки;
- перезапустить только отдельную копию, не текущий Codex.

Результат: веб становится рабочим инструментом, а не макетом.

### 6. `WEB_DESIGN_FINISH_PASS`

Цель: доделать внешний вид только после того, как основные сценарии работают.

Зачем: дизайн должен обслуживать рабочий продукт, а не скрывать нестабильную
логику.

Делаем:

- навигацию;
- нормальные состояния кнопок;
- пустые состояния;
- ошибки;
- компактные журналы;
- понятную главную панель.

Результат: веб выглядит и ощущается как готовый продуктовый слой.

### 7. `DESKTOP_APP_PORT_PASS`

Цель: перенести проверенные веб-сценарии в настольное приложение.

Зачем: не изобретать заново, а перенести только то, что уже работает.

Делаем:

- те же экраны;
- те же действия;
- те же проверки;
- те же ограничения безопасности;
- без новых крупных функций.

Результат: настольное приложение повторяет проверенную веб-логику.

### 8. `DESKTOP_APP_PACKAGE_PASS`

Цель: собрать безопасный пакет настольного приложения.

Зачем: получить продукт, который можно передавать и ставить.

Проверяем:

- нет токенов;
- нет auth-файлов;
- нет runtime state;
- нет логов и дампов;
- приложение запускается;
- базовый сценарий проходит.

Результат: готовая безопасная сборка.

## Стоп-правила

Останавливаемся только по делу:

- тест упал;
- браузерная проверка не прошла;
- команда дала противоречивый результат;
- есть риск затронуть текущий Codex;
- есть риск записать личные данные в репозиторий;
- действие требует живой мутации без понятного отката;
- UI показывает зелёное без доказательства.

Не останавливаемся ради:

- красивой теории;
- полного аудита всего репозитория;
- переписывания старых документов;
- улучшения соседнего слоя;
- повторного доказательства того, что уже проверено.

## Минимальная проверка контура

Для маленького безопасного контура:

- целевой тест;
- браузерная проверка, если есть UI;
- `git diff --check`;
- commit;
- push.

Для рискованного контура:

- целевые тесты;
- проверка JSON-пакетов, если они есть;
- один независимый аудит;
- closeout;
- commit;
- push.

## Текущий следующий шаг

Следующий контур:

`RUNTIME_LAUNCHER_OWNER_PROCEDURE_SERIALIZATION_FIX_PASS`

После него:

`WEB_SAFE_APP_COPY_LAUNCH_PASS`

Это самый короткий путь к практическому продукту:

1. убрать текущий технический блокер;
2. дать вебу безопасную кнопку запуска отдельной копии;
3. проверить это браузером;
4. затем идти к аккаунтам и остальным функциям.
